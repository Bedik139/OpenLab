3. HOW AUTHENTICATION WORKS
================================================================================

- All user accounts are stored in localStorage under key 'openlab_accounts'
- On first load, SAMPLE_USERS (6 users) are seeded into this store
- Login validates email + password against stored accounts
- On success, a session object (without password) is saved to 'openlab_user'
- Registration creates a new entry in the accounts store with the password
- Duplicate emails and student IDs are rejected during registration
- After registration, the user is auto-logged in and redirected to dashboard
- Logging out removes the session and redirects to the landing page
- Password changes update the account in the accounts store directly

  - profile.html

TECHNICIAN-ONLY PAGE (students get "Access denied"):

REGISTRATION

  - Validates: password match, min 8 chars, 8-digit student ID
  - Rejects duplicate email or student ID

  - Seat map with available/occupied/reserved/selected states

  - Enter student ID, select lab, seat, date, time
  - Validates student ID format and seat format
  - Looks up student name from accounts if registered
  - Saves to global reservation store with isWalkIn flag
  - Right panel shows current walk-in reservations from storage
  - Remove no-show button cancels the reservation in storage

DASHBOARD
  - Dynamic stats computed from actual reservation data
  - Upcoming reservations list
  - Quick actions (Reserve, Manage, Walk-In for techs, Profile, Find Users)
  - Recent activity feed
  - Favorite labs section
  - Technician dashboard shows all-user data

PROFILE
  - View and edit personal information
  - Statistics computed from reservation data
  - Change password (validates current password)
  - Change avatar (file upload, max 2MB)
  - Email notification toggle
  - Delete account (removes from all stores)
  - Role badge shows Student or Technician

SEARCH
  - Global search bar in navbar
  - Lab search on available slots page
  - User search by name, ID, or college filter

11. BENEDICT'S BACKEND CONTRIBUTIONS

A. JSON AND FETCH
-----------------
Replaced all localStorage reads/writes in the frontend with fetch() calls
to the backend REST API. The frontend no longer stores or reads data from
localStorage directly — all data flows through HTTP requests to the
Node.js server.

Key changes to js/app.js:
  - authenticateUser() now sends POST /api/login with JSON body
    {email, password} and receives a session cookie on success.
  - registerAccount() now sends POST /api/register with JSON body
    containing all registration fields.
  - getReservations() now sends GET /api/reservations and parses
    the JSON response to populate the reservations page.
  - saveReservation() now sends POST /api/reservations with the
    reservation object as JSON.
  - cancelReservation() sends DELETE /api/reservations/:id.
  - editReservation() sends PUT /api/reservations/:id with updated fields.
  - getAllReservations() (technician) sends GET /api/reservations/all.
  - updateProfile() sends PUT /api/profile with updated user fields.
  - changePassword() sends PUT /api/profile/password with
    {currentPassword, newPassword}.
  - deleteAccount() sends DELETE /api/profile.
  - Walk-in endpoints: POST /api/walkin, GET /api/walkin,
    DELETE /api/walkin/:id.
  - All fetch calls include { credentials: 'include' } to send session
    cookies with every request.
  - Error responses from the server (4xx/5xx) are caught and displayed
    to the user via the existing alert/toast UI.

C. COOKIES AND SESSIONS
------------------------
Implemented server-side session management using express-session with
a MongoDB session store (connect-mongo).

Setup (in server.js / app.js on the backend):
  - express-session configured with:
      secret: environment variable SESSION_SECRET
      resave: false
      saveUninitialized: false
      cookie: { maxAge: 1000*60*60*24*7 } (7-day sessions)
      store: MongoStore.create({ mongoUrl: MONGO_URI })
  - On successful login, the user object (without password) is stored
    in req.session.user.
  - authMiddleware checks req.session.user on protected routes and
    redirects to /login if absent.
  - techMiddleware additionally checks req.session.user.accountType
    === 'technician' for technician-only routes.
  - Logout destroys the session via req.session.destroy() and clears
    the cookie.
  - "Remember Me" checkbox sets a longer maxAge (21 days) on the
    session cookie.

Session-protected routes:
  GET  /dashboard          → authMiddleware
  GET  /reservations       → authMiddleware
  GET  /reserve            → authMiddleware
  GET  /profile            → authMiddleware
  PUT  /api/profile        → authMiddleware
  POST /api/reservations   → authMiddleware
  GET  /walkin             → authMiddleware + techMiddleware
  POST /api/walkin         → authMiddleware + techMiddleware
